Problem Statement & Hypothesis

Problem Statement

Develop the heart rate sensor pipeline using base heart rate and HRV and build a history context using external factors to resolve ambiguity in sensor data to safely presume fatigue/drowsiness.

Hypothesis

  • Using LLMs to predict the Karolinska-Sleepiness Scale based on time of day, duration of drive, SD in Lane Position/Speed, and history context can increase reasoning accuracy.
  • An LLM's accuracy will be higher when provided with a structured history of prior predictions and sensor summaries compared to a stateless LLM.
  • Summarized historical context will achieve similar or better accuracy than raw full-history prompts while reducing token usage and latency.
  • Models with conflict/reliability context will recover baseline accuracy faster after simulated sensor dropout compared to stateless LLMs.

Original Data Analysis: Correlations

Key Findings

  • `heartRate` vs `drowsiness` (-0.68): Strongest correlation. This is a key finding, suggesting that as heart rate *decreases*, drowsiness *increases*.
  • `ppgRed` vs `ppgIR` (0.77): Very strong positive correlation, as expected since both are infrared/near-infrared light sensors.
  • `ppgRed`/`ppgIR` vs `drowsiness` (-0.39 / -0.41): Moderate negative correlation, likely linked to changes in blood volume/flow associated with heart rate.

PCA Cumulative Explained Variance

The first component explains 35% of the variance. We need ~25-30 components to capture 95% of the variance, indicating high dimensionality.

2D PCA Visualization

Visualizing the first two components shows some clustering, but the classes (0.0, 1.0, 2.0) are heavily overlapped. This suggests a simple linear model would perform poorly.

Model Comparison: Confusion Matrices

Random Forest

  • Strong on Class 0.0 (Alert) and 2.0 (Drowsy).
  • Struggles with Class 1.0, frequently misclassifying it as 0.0 (9 instances).

XGBoost

  • Similar struggle with Class 1.0 (11 misclassified as 0.0).
  • Slightly worse on Class 0.0 but perfect on Class 2.0 in this test set.

Many HRV features are heavily right-skewed (e.g., `HRV_RMSSD`), while others (`HRV_DFA_alpha1`) are more normally distributed. This supports the need for scaling (like StandardScaler used before PCA).
